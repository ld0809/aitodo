# 项目基本知识维护指南

> 目标：确保 `docs/PROJECT_KNOWLEDGE.md` 在功能演进时及时更新，并在开发时被充分使用。

## 开发前检查

1. 先读 `AGENTS.md`。
2. 再读 `docs/PROJECT_KNOWLEDGE.md`。
3. 如果任务涉及特定阶段或历史背景，再读 `docs/PRD.md`、`docs/ARCHITECTURE.md` 或相关阶段文档。
4. 如果发现知识文档与代码不一致，以代码为准，并把文档修正纳入本次变更。

## 变更后检查

每次代码变更后都要判断是否触发知识更新：

- 架构：模块、目录、实体、插件、认证、响应格式、迁移策略。
- 业务：待办、卡片、共享、TAPD、AI 报告、小程序、OpenClaw 的规则。
- UI：首页视图、卡片操作、列表模式、弹窗流程、共享组件和设计 token。
- API：路径、DTO、响应字段、错误规则、权限边界。

如果触发，更新 `docs/PROJECT_KNOWLEDGE.md`。如果没有触发，在最终说明中写明“本次未改变基本知识”。

## 推荐流程

1. 用 `git diff --name-only` 找出本次改动范围。
2. 对照 `docs/PROJECT_KNOWLEDGE.md` 搜索相关章节。
3. 修改文档，只记录稳定事实，不记录临时调试过程。
4. 如果涉及数据库结构，确认 migration 与实体描述同步。
5. 如果涉及 UI，确认交互说明与实际页面一致。
6. 在最终交付中列出文档更新情况。

## 文档职责边界

- `docs/PROJECT_KNOWLEDGE.md`：当前代码事实和开发必读知识。
- `docs/PRD.md`：产品阶段规划和历史需求。
- `docs/ARCHITECTURE.md`：偏整体架构方案，可能包含历史设计。
- 阶段文档：某个阶段的方案、部署或验收背景。

当历史文档与当前代码不一致时，不要直接删除历史上下文；优先在 `PROJECT_KNOWLEDGE.md` 中记录当前事实。
